# 单兵边缘智能小脑服务器计划书

## 1. 项目概述

单兵边缘智能小脑服务器是一款面向警务单兵装备场景的便携式边缘 AI 计算终端。设备独立于安卓手机和执法记录仪运行，内部部署小型 Linux 系统、视觉识别算法、本地大模型和安全存储模块，可在无公网或弱网环境下完成车牌识别、人脸识别、视频摘要、执法日报生成和移动端算力协同。

该设备的定位不是普通执法记录仪本体，而是执法记录仪大小或略大的边缘 AI 服务器。它可以挂载在胸前、肩部、腰带或装备包上，通过 Wi-Fi Direct、局域网、USB-C 或专用短距通信方式与手机、智能头盔、执法记录仪和后台系统协同。

## 2. 建设目标

### 2.1 核心目标

1. 在本地完成车牌识别、人脸检测与候选比对，减少对云端算力的依赖。
2. 对执法记录视频进行本地抽帧、事件识别、语音转写和摘要生成。
3. 基于当天巡逻记录、识别结果、语音文本和人工补充信息，自动生成日常工作报告。
4. 为手机端和安卓设备提供近端 AI 算力服务，提高响应速度并降低公网依赖。
5. 支持完全离线工作，回到内网或指挥中心后再同步数据。
6. 满足警用装备对安全、审计、加密和证据链完整性的要求。

### 2.2 产品形态目标

| 指标 | 建议目标 |
|---|---|
| 设备形态 | 单兵携带式边缘 AI 服务器 |
| 尺寸目标 | 约 120 x 80 x 30-40mm，可根据散热调整 |
| 重量目标 | 400-800g |
| 佩戴方式 | 胸挂、肩挂、腰带挂载、装备包固定 |
| 工作系统 | 小型 Linux 系统 |
| 工作模式 | 离线独立运行 + 手机协同 + 内网同步 |
| 防护能力 | IP54 起步，警用版本建议 IP65 |
| 续航目标 | 混合工作 4-6 小时，高负载分析 2-3 小时 |

## 3. 使用场景

### 3.1 巡逻执勤

单兵携带小脑服务器，执法记录仪或智能头盔持续采集视频。小脑对视频流进行车牌、人脸、人员、车辆和异常行为检测，并把关键结果推送给手机端显示。

### 3.2 现场执法

执法过程中，小脑自动记录关键时间点、地点、视频片段、识别对象和语音内容。现场执法结束后，系统生成结构化事件摘要，辅助民警快速整理执法记录。

### 3.3 弱网或无网环境

在地下空间、山区、灾害现场、大型活动安保等网络不稳定场景下，小脑可完全本地运行。所有识别、摘要和报告先存储在本机，待恢复网络后再同步到警务平台。

### 3.4 手机端算力扩展

手机端需要分析图片、短视频、语音或现场文本时，可以优先连接小脑进行本地推理。相比远程调用云端算力，响应更快，且敏感数据不必立即出公网。

### 3.5 班后汇总

巡逻结束后，小脑根据当天视频、音频、定位、人工标注和识别结果生成巡逻日报、异常事件清单、交接班说明和重点视频摘要。

## 4. 功能设计

### 4.1 视频接入模块

| 功能 | 说明 |
|---|---|
| 实时视频流接入 | 支持执法记录仪、智能头盔、手机摄像头、车载摄像头 |
| 文件导入 | 支持手机或记录仪上传短视频、图片和音频 |
| 协议支持 | RTSP、USB 摄像头、Wi-Fi Direct、局域网 HTTP/gRPC |
| 视频预处理 | 抽帧、降噪、去抖、亮度增强、夜间增强 |
| 关键片段缓存 | 自动缓存识别到异常事件前后的短视频片段 |

### 4.2 车牌识别模块

| 功能 | 说明 |
|---|---|
| 车牌检测 | 检测视频帧中的车辆和车牌区域 |
| 车牌矫正 | 对倾斜、模糊、远距离车牌进行几何矫正 |
| 车牌 OCR | 识别普通蓝牌、黄牌、新能源牌、警用牌等 |
| 多帧融合 | 多帧结果投票，提高移动场景识别准确率 |
| 置信度输出 | 每条识别结果附带置信度和原始帧证据 |
| 本地黑白名单 | 支持本地布控库、关注车辆库和临时名单 |

### 4.3 人脸识别模块

| 功能 | 说明 |
|---|---|
| 人脸检测 | 实时检测画面中的人脸区域 |
| 质量评分 | 判断清晰度、角度、遮挡、光照是否满足识别要求 |
| 特征提取 | 生成本地人脸特征向量 |
| 1:N 比对 | 与本地授权人员库、重点关注库进行候选比对 |
| 候选提示 | 输出候选人员、相似度、证据帧，不直接作为最终结论 |
| 隐私控制 | 非授权场景可只做检测和模糊化，不做人脸库比对 |

### 4.4 视频摘要模块

视频摘要不建议直接把完整视频输入大模型，而应采用“结构化事件 + 关键帧 + 语音文本 + 大模型生成”的方式。

处理流程：

1. 视频抽帧，按时间和画面变化提取关键帧。
2. 视觉算法识别人、车、车牌、人脸、动作和场景。
3. 本地 ASR 将语音转为文字。
4. 事件引擎生成结构化时间线。
5. 本地大模型基于结构化数据生成摘要。
6. 输出短摘要、详细摘要、事件列表和证据片段索引。

输出内容：

| 输出项 | 说明 |
|---|---|
| 视频一句话摘要 | 适合列表快速浏览 |
| 时间线摘要 | 按时间顺序列出关键事件 |
| 异常事件清单 | 提取争执、追踪、盘查、车辆异常等事件 |
| 证据片段索引 | 标记每个事件对应的视频时间段 |
| 人工确认状态 | 支持民警确认、修改、驳回 AI 生成内容 |

### 4.5 日常报告生成模块

本地大模型主要用于报告生成、摘要整理和结构化文本输出，不承担法律结论判断。

可生成内容：

1. 巡逻日报。
2. 执法过程摘要。
3. 交接班说明。
4. 重点人员车辆出现记录。
5. 异常事件报告。
6. 视频摘要说明。
7. 工作量统计说明。

报告字段建议：

| 字段 | 说明 |
|---|---|
| 执勤人员 | 从登录账号或设备绑定关系获取 |
| 执勤时间 | 自动读取设备时间 |
| 执勤区域 | GPS/北斗定位或人工选择 |
| 主要事件 | 从事件时间线自动生成 |
| 车辆识别情况 | 汇总车牌识别结果 |
| 人脸候选提示 | 只记录候选和置信度，不写绝对结论 |
| 异常情况 | 需人工确认后纳入正式报告 |
| 附件索引 | 对应视频片段、图片、音频和日志 |

### 4.6 手机协同模块

| 功能 | 说明 |
|---|---|
| 设备发现 | 手机自动发现附近小脑设备 |
| 安全配对 | 二维码、证书或警员账号绑定 |
| 任务提交 | 手机上传图片、短视频、音频或文本分析任务 |
| 结果返回 | 小脑返回识别结果、摘要和报告草稿 |
| 状态监控 | 手机显示电量、温度、存储、模型状态和网络状态 |
| 人工确认 | 民警在手机端确认或修改 AI 结果 |

### 4.7 安全与审计模块

| 功能 | 说明 |
|---|---|
| 磁盘加密 | 本地视频、图片、报告和特征库加密存储 |
| Secure Boot | 防止系统被替换或篡改 |
| TPM/安全芯片 | 存储密钥、设备证书和启动度量信息 |
| 权限控制 | 区分普通警员、管理员、审计员、维护人员权限 |
| 操作审计 | 记录查看、导出、删除、同步、模型调用等操作 |
| 远程锁定 | 设备丢失后可远程禁用和擦除敏感数据 |
| 数据脱敏 | 非授权场景下对人脸、车牌和个人信息脱敏显示 |

## 5. 硬件配置方案

### 5.1 推荐方案：均衡型单兵小脑

该方案适合首版产品和工程样机。

| 部件 | 推荐配置 |
|---|---|
| 主控平台 | NVIDIA Jetson Orin NX 16GB |
| AI 算力 | 最高约 157 TOPS，10W-40W 可调 |
| CPU | 8 核 Arm Cortex-A78AE |
| GPU | NVIDIA Ampere 架构 GPU |
| 内存 | 16GB LPDDR5 |
| 系统存储 | 64GB 或 128GB eMMC |
| 数据存储 | 1TB NVMe SSD，预留 2TB 版本 |
| 无线通信 | Wi-Fi 6/6E、蓝牙 |
| 蜂窝网络 | 可选 4G/5G 模块 |
| 定位 | GPS/北斗/GNSS |
| 安全芯片 | TPM 2.0 或国产安全芯片 |
| 接口 | USB-C PD、USB 3.0、MIPI/USB 摄像头、调试串口 |
| 电池 | 60Wh 推荐，40Wh 为最低可用配置 |
| 散热 | 铝合金外壳 + 导热片 + 主动风扇 |
| 防护 | IP54 起步，警用版本建议 IP65 |

### 5.2 低功耗视觉版

适合只做车牌、人脸和目标识别，不强调本地大模型生成能力的版本。

| 部件 | 推荐配置 |
|---|---|
| 主控平台 | RK3588 / RK3576 |
| AI 加速 | 内置 6 TOPS NPU，可选 Hailo-8 26 TOPS |
| 内存 | 8GB-16GB |
| 存储 | 256GB-1TB |
| 优点 | 成本低、功耗低、体积小 |
| 不足 | 本地大模型能力弱，生态和部署复杂度高于 Jetson |

### 5.3 高性能背包版

适合多路视频、高级图文理解、大模型推理和车载/指挥场景。

| 部件 | 推荐配置 |
|---|---|
| 主控平台 | Jetson AGX Orin 32GB/64GB |
| AI 算力 | 最高约 275 TOPS |
| 内存 | 32GB-64GB |
| 存储 | 2TB NVMe SSD |
| 优点 | 性能强，可运行更复杂模型 |
| 不足 | 功耗高、散热压力大、体积和成本较高 |

### 5.4 方案对比

| 方案 | 性能 | 功耗 | 体积 | 成本 | 推荐程度 |
|---|---|---|---|---|---|
| RK3588 | 中低 | 低 | 小 | 低 | 适合低配版 |
| RK3588 + Hailo-8 | 中 | 低 | 小 | 中 | 适合视觉识别版 |
| Jetson Orin Nano 8GB | 中 | 中低 | 小 | 中 | 可做轻量验证 |
| Jetson Orin NX 16GB | 高 | 中 | 中 | 中高 | 首版推荐 |
| Jetson AGX Orin 32/64GB | 很高 | 高 | 大 | 高 | 车载或背包版 |

## 6. 模型配置建议

### 6.1 大语言模型

本地大模型建议以 Qwen3.5-4B 作为首版主力模型，使用 INT4 或 INT8 量化部署。该模型用于日常报告生成、视频摘要、事件时间线整理、手机端文本分析和短视频分析结果归纳。Qwen3.5-2B 作为低功耗备用模型，Qwen3.5-9B 仅作为充电或空闲状态下的高质量批处理模型，不作为实时巡逻阶段默认模型。

| 模型 | 参数规模 | 适合任务 | 建议 |
|---|---|---|---|
| Qwen3.5-4B | 约 4B | 中文报告、视频摘要、事件整理、图文理解 | 首版主力模型 |
| Qwen3.5-2B | 约 2B | 快速摘要、模板日报、设备助手 | 低功耗备用模型 |
| Qwen3.5-9B | 约 9B | 高质量总结、复杂报告润色、深度分析 | 仅限空闲/充电批处理 |
| Qwen3.5-0.8B | 约 0.8B | 指令分类、简单问答、状态解释 | 超低功耗辅助模型 |

### 6.1.1 Qwen3.5-4B 运行策略

| 项目 | 设计要求 |
|---|---|
| 部署格式 | INT4/Q4 量化优先，保留 INT8 版本用于精度对比 |
| 推荐上下文 | 默认 8K-16K tokens，最多谨慎扩展到 32K tokens |
| 运行阶段 | 现场短摘要、班后日报、事件整理、手机端近端分析 |
| 实时策略 | 不与高负载视频分析长期并发，实时巡逻阶段按任务触发调用 |
| 输出控制 | 默认限制 thinking 输出和最大生成长度，防止长推理拖慢设备 |
| 输入方式 | 优先输入结构化事件、ASR 文本、关键帧描述和人工补充信息 |
| 回退策略 | 温度过高、内存紧张或电量不足时切换到 Qwen3.5-2B |

### 6.2 视觉模型

| 任务 | 模型方向 |
|---|---|
| 车牌检测 | YOLO 系列轻量模型、PP-YOLOE、定制车牌检测模型 |
| 车牌 OCR | CRNN、SVTR、PaddleOCR 轻量化模型 |
| 人脸检测 | RetinaFace、SCRFD、YOLO-Face |
| 人脸特征 | ArcFace、MobileFaceNet、InsightFace 轻量模型 |
| 目标检测 | YOLOv8/YOLOv10/RT-DETR 轻量版本 |
| 语音识别 | Whisper tiny/base、本地中文 ASR 小模型 |

### 6.3 模型部署策略

1. 视觉模型优先使用 TensorRT 或 ONNX Runtime 加速。
2. 大语言模型优先使用 llama.cpp、Ollama 或 TensorRT-LLM 轻量部署。
3. 报告生成模型以 Qwen3.5-4B INT4/Q4 量化版本为默认配置，控制显存、内存和功耗。
4. 视频摘要采用批处理，不在巡逻实时阶段持续运行大模型。
5. 建立模型版本管理，确保每次识别和报告生成可追溯。
6. 在 Jetson Orin NX 16GB 上运行时，应限制上下文窗口，不使用 Qwen3.5 标称超长上下文作为默认配置。
7. 高质量总结任务可在充电、休息或回站点后调用 Qwen3.5-9B，但不得影响实时车牌、人脸和视频采集任务。

## 7. 软件架构

### 7.1 系统架构

```text
手机 App / 安卓设备 / 执法记录仪
        |
        | Wi-Fi Direct / USB-C / 局域网
        |
单兵边缘智能小脑服务器
        |
        |-- 视频接入服务
        |-- 车牌识别服务
        |-- 人脸识别服务
        |-- 语音转写服务
        |-- 事件时间线服务
        |-- 本地大模型服务
        |-- 报告生成服务
        |-- 安全审计服务
        |-- 数据同步服务
        |
警务后台 / 指挥平台 / 证据管理系统
```

### 7.2 推荐软件栈

| 层级 | 技术选型 |
|---|---|
| 操作系统 | Ubuntu / JetPack Linux |
| 容器化 | Docker / Docker Compose |
| 视频处理 | GStreamer / FFmpeg |

| 同步机制 | 断点续传、任务队列、后台同步 |
| 日志审计 | 本地审计日志 + 后台汇总 |

### 7.4 Linux 系统裁剪与加固需求

单兵边缘智能小脑服务器不应直接使用完整桌面版 Linux 作为量产系统。系统应按嵌入式设备方式进行裁剪和加固，目标是保留 AI 推理、视频处理、网络通信、安全审计、设备管理和 OTA 升级能力，移除与警务执法业务无关的软件包、服务和调试入口，降低功耗、启动时间、存储占用和安全攻击面。

#### 7.4.1 系统版本分层

| 镜像类型 | 用途 | 特点 |
|---|---|---|
| 开发调试版 | PoC、算法调试、性能分析 | 保留 SSH、包管理器、编译工具、调试工具和性能分析工具 |
| 工程测试版 | 工程样机、外场试用 | 精简桌面组件，保留有限调试入口，启用基础安全策略 |
| 量产执法版 | 正式警用交付 | 只读系统分区、签名 OTA、关闭默认调试入口、强制权限和审计 |

#### 7.4.2 必须保留的系统能力

| 能力 | 说明 |
|---|---|
| AI 推理运行时 | Jetson 保留 CUDA、TensorRT、ONNX Runtime、llama.cpp；国产平台保留 CANN、TPU SDK、NPU SDK 等 |
| 视频处理能力 | 保留 GStreamer、FFmpeg、硬件编解码驱动、摄像头驱动 |
| 网络通信能力 | 保留 Wi-Fi Direct、热点模式、蓝牙配对、HTTP/gRPC API、内网同步 |
| 安全能力 | 保留磁盘加密、Secure Boot、TPM/安全芯片、证书管理、签名校验 |
| 设备管理能力 | 保留电池、电源、温度、风扇、存储、网络、看门狗管理 |
| OTA 升级能力 | 支持系统、应用、模型、配置和证书的签名升级与回滚 |
| 日志审计能力 | 保留系统日志、安全日志、模型调用日志、数据导出日志和异常日志 |
| 服务编排能力 | 保留 systemd；如采用容器化部署，保留 Docker 或 containerd |

#### 7.4.3 应裁剪或关闭的组件

| 类别 | 裁剪要求 |
|---|---|
| 图形桌面 | 量产版本去掉 GNOME、KDE 等完整桌面，只保留命令行或轻量 Web 管理页 |
| 办公和娱乐软件 | 去掉浏览器、办公套件、播放器、游戏、示例程序等无关软件 |
| 编译开发工具 | 量产版本去掉 gcc、make、cmake、gdb、strace 等工具，开发版保留 |
| 包管理能力 | 量产版本限制 apt/yum 直接安装，统一走签名 OTA |
| 无关系统服务 | 关闭打印服务、邮件服务、Samba、Avahi、无关发现服务和演示服务 |
| 远程登录 | SSH 默认关闭，仅维护模式或授权证书下临时开启 |
| USB 自动挂载 | 禁止普通 U 盘自动挂载，只允许授权采集设备和维护设备 |
| 蓝牙功能 | 仅保留配对和必要通信能力，关闭文件传输等非业务能力 |
| 内核模块 | 移除不需要的驱动和文件系统模块，减少攻击面 |
| 默认账号 | 删除默认用户，禁止 root 直接登录，服务使用最小权限账号运行 |

#### 7.4.4 量产系统加固要求

1. 系统分区采用只读挂载，业务数据、日志和模型文件放在独立数据分区。
2. 数据分区加密，设备证书和密钥绑定 TPM 或安全芯片。
3. 启用 Secure Boot，禁止未签名内核、驱动、系统镜像和启动脚本运行。
4. OTA 包必须签名校验，升级失败必须支持自动回滚。
5. 模型文件、算法包和配置文件必须进行版本管理和签名校验。
6. 服务按最小权限运行，不允许业务服务长期使用 root 权限。
7. 关闭默认密码、默认开放端口和无鉴权管理接口。
8. 对查看、导出、删除、同步、模型调用、远程维护等操作记录审计日志。
9. 设备异常断电后，系统必须能自动恢复，不损坏原始视频和事件索引。
10. 支持丢失设备远程锁定、证书吊销和敏感数据擦除。

#### 7.4.5 推荐系统路线

| 阶段 | 推荐系统 |
|---|---|
| PoC 样机 | Ubuntu / JetPack 原版，优先保证算法跑通 |
| 工程样机 | 精简 Ubuntu Server / Debian / JetPack Runtime |
| 量产版本 | Yocto / Buildroot / openEuler Embedded / 定制 Debian |
| 国产化版本 | openEuler Embedded / 麒麟嵌入式 / 统信嵌入式，按项目国产化要求选择 |

#### 7.4.6 验收指标

| 指标 | 建议要求 |
|---|---|
| 启动时间 | 冷启动进入服务可用状态建议小于 60 秒 |
| 默认开放端口 | 只开放业务必需端口，维护端口默认关闭 |
| 系统分区 | 量产版只读，异常断电后可恢复 |
| OTA | 支持签名校验、断点恢复、失败回滚 |
| 日志 | 安全审计日志不可被普通用户删除或篡改 |
| 调试入口 | 量产版默认关闭 SSH、串口 shell 和未授权 ADB 类入口 |
| 软件包数量 | 去除无业务用途的软件包，形成可复现系统镜像清单 |
| 安全扫描 | 量产前完成端口扫描、弱口令检查、依赖漏洞检查和权限检查 |

## 9. 续航与功耗设计

### 9.1 功耗估算

| 工作模式 | 功耗估计 | 40Wh 电池 | 80Wh 电池 |
|---|---:|---:|---:|
| 待机/轻负载 | 6-10W | 4-6 小时 | 8-12 小时 |
| 单路视频识别 | 12-20W | 2-3 小时 | 4-6 小时 |
| 视频识别 + 摘要生成 | 25-40W | 1-1.5 小时 | 2-3 小时 |
| 高负载多任务 | 40W 以上 | 不推荐 | 1-2 小时 |

### 9.2 功耗控制策略

1. 巡逻实时阶段只运行必要的车牌、人脸和目标检测模型。
2. 大模型报告生成放到低峰、休息或充电时批处理。
3. 支持性能档位：省电、标准、高性能、充电增强。
4. 温度过高时自动降频，并暂停非关键 AI 任务。
5. 支持外接电池或热插拔电池，满足长时间执勤。

## 10. 安全合规设计

### 10.1 数据安全

1. 本地磁盘全盘或分区加密。
2. 视频、图片、人脸特征、车牌记录单独加密。
3. 设备证书绑定硬件安全芯片。
4. 关键数据导出必须经过授权和审计。
5. 支持设备丢失后的远程锁定和数据擦除。

### 10.2 算法合规

1. 人脸识别结果只作为候选提示，不作为自动执法结论。
2. 所有 AI 生成报告必须支持人工确认和修改。
3. 识别结果必须保留置信度、证据帧和模型版本。
4. 模型更新必须记录版本、发布时间、适用范围和回滚方案。
5. 对非授权人员、非必要场景支持隐私脱敏。

### 10.3 证据链保护

1. 原始视频不可被 AI 处理流程覆盖。
2. AI 生成内容与原始证据分开存储。
3. 对原始媒体文件生成哈希值。
4. 记录所有查看、分析、导出和删除操作。
5. 与后台同步时采用签名和完整性校验。

## 12. 首版 MVP 建议

第一版不要追求所有能力一次到位，建议聚焦以下四项能力：

1. 手机、智能头盔或执法记录仪视频接入。
2. 车牌识别和人脸候选提示。
3. 当日事件时间线自动生成。
4. 巡逻日报和视频摘要自动生成。

首版推荐硬件为 Jetson Orin NX 16GB。该方案在性能、体积、功耗、开发生态和产品化风险之间最平衡。Jetson AGX Orin 适合后续车载版或背包高配版；RK3588 + Hailo-8 适合后续低成本视觉识别版。

首版本地大模型确定为 Qwen3.5-4B INT4/Q4 量化版本。Qwen3.5-4B 作为日报生成、视频摘要和事件整理的主力模型；Qwen3.5-2B 作为低功耗备用；Qwen3.5-9B 作为非实时高质量批处理模型，不进入实时巡逻默认链路。

## 13. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 散热压力大 | 降频、死机、佩戴不适 | 主动散热、铝合金外壳、功耗档位 |
| 续航不足 | 无法覆盖完整执勤 | 大电池、外接电池、任务调度 |
| 视频摘要效果不稳定 | 报告质量不一致 | 使用结构化事件驱动报告生成 |
| 人脸识别合规风险 | 法务和隐私风险 | 候选提示、人工确认、权限控制 |
| 弱网同步失败 | 数据无法及时上传 | 断点续传、本地缓存、同步重试 |
| 算法误识别 | 误导现场判断 | 置信度、证据帧、人工确认 |
| 成本过高 | 难以规模化 | 区分标准版、低配版、高配版 |
| Linux 系统未裁剪 | 启动慢、功耗高、攻击面大、维护不可控 | 建立开发版、工程版、量产版三类镜像，量产版只保留业务必需能力 |
| 调试入口残留 | 设备被非法登录、数据泄露、证据链受损 | 量产版默认关闭 SSH、串口 shell、USB 自动挂载和未授权维护接口 |

## 14. 建议结论

该项目具备落地价值，尤其适合弱网、移动执法、现场快速分析和班后报告自动生成场景。

建议采用“边缘智能服务器”路线，而不是把所有能力塞进普通执法记录仪内部。首版使用 Jetson Orin NX 16GB 做工程样机，先验证车牌识别、人脸候选提示、事件时间线和本地报告生成。等算法、续航、散热和业务流程验证稳定后，再根据实际场景拆分为标准单兵版、低功耗视觉版和高性能背包版。
